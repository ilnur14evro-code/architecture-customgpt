# Архитектура Custom GPT для написания постов

## Назначение

Custom GPT предназначен только для одной задачи — написания качественных постов для социальных сетей.

Архитектура не является универсальным фреймворком и не предназначена для других типов задач.

---

# Структура проекта

```text
SYSTEM.md

ARCHITECTURE.md

ORCHESTRATOR.md

WORKFLOW_WRITE_POST.md

ROLE_WRITER.md

ROLE_CRITIC.md

POST_STANDARD.md

OUTPUT_FORMAT.md
```

---

# Назначение компонентов

## SYSTEM

Определяет общие правила работы GPT:

- язык общения;
- общие ограничения;
- правила поведения модели.

SYSTEM не участвует в процессе написания постов.

---

## ARCHITECTURE

Описывает архитектуру проекта.

Определяет:

- назначение компонентов;
- ответственность компонентов;
- взаимодействие между ними.

---

## ORCHESTRATOR

Отвечает только за управление выполнением процесса.

### Алгоритм работы

1. Получить запрос пользователя.
2. Запустить `WORKFLOW_WRITE_POST`.
3. Передать Workflow:
   - запрос пользователя.
4. Получить результат от Workflow.
5. Передать результат в `OUTPUT_FORMAT`.

ORCHESTRATOR никогда:

- не пишет текст;
- не проверяет качество;
- не принимает решений о содержании поста.

---

## WORKFLOW_WRITE_POST

Полностью управляет процессом создания поста.

### Алгоритм

1. Запустить Writer.
2. Writer создает первую версию поста.
3. Critic проверяет текст.
4. Если Critic вернул статус `REVISION_REQUIRED` — передать Writer замечания Critic.
5. Writer дорабатывает текст на основании замечаний Critic.
6. Повторять цикл Writer ⇄ Critic.
7. Если Critic вернул статус `APPROVED` — завершить Workflow и вернуть результат ORCHESTRATOR.
8. Если достигнуто максимальное количество итераций (`MAX_ITERATIONS = 5`) до получения статуса `APPROVED` — завершить Workflow с последней версией текста.

### Completion Criteria

Workflow считается завершенным после:

- получения статуса `APPROVED` от Critic;

или

- достижения максимального количества итераций (`MAX_ITERATIONS = 5`).

Получение статуса `APPROVED` считается успешным завершением Workflow.

---

## ROLE_WRITER

Отвечает только за создание и доработку текста.

### Input

- запрос пользователя;
- POST_STANDARD;
- замечания Critic (если есть).

### Алгоритм

Если замечаний нет:

- создать первую версию поста.

Если замечания есть:

- исправить только указанные замечания;
- сохранить удачные части текста;
- не переписывать пост полностью без необходимости.

При создании поста Writer:

- соблюдает требования POST_STANDARD;
- самостоятельно адаптирует текст под платформу, если она указана пользователем;
- создает универсальный пост, если платформа не указана.

Writer не проверяет качество и не принимает решений о завершении Workflow.

### Output

- текущая версия поста.

---

## ROLE_CRITIC

Отвечает только за проверку текста.

### Input

- текущая версия поста;
- POST_STANDARD;
- запрос пользователя.

### Алгоритм

Проверить соответствие поста:

- требованиям POST_STANDARD;
- особенностям платформы, если она указана пользователем;
- требованиям к универсальному посту, если платформа не указана.

Если требования нарушены:

Вернуть

**Status:**

`REVISION_REQUIRED`

**Revision Feedback:**

- список замечаний;
- рекомендации по исправлению.

Если требования выполнены:

Вернуть

**Status:**

`APPROVED`

Critic никогда:

- не изменяет текст;
- не пишет новую версию поста.

### Output

**Status:**

- `APPROVED`
- `REVISION_REQUIRED`

**Revision Feedback:**

- список замечаний и рекомендаций (при наличии).

---

## POST_STANDARD

Главный файл проекта.

Единственный источник требований к качеству постов.

Определяет:

- требования к ценности;
- требования к содержанию;
- требования к стилю;
- требования к читаемости;
- требования к достоверности;
- требования к качеству текста.

POST_STANDARD не содержит требований конкретных социальных сетей.

---

## OUTPUT_FORMAT

Получает утвержденный результат от ORCHESTRATOR.

Отвечает только за отображение результата пользователю.

OUTPUT_FORMAT:

- выводит результат без изменений;
- не изменяет текст;
- не сокращает текст;
- не адаптирует пост под платформу;
- не проверяет качество;
- не принимает решений.

# Логика работы

```text
User
 │
 ▼
ORCHESTRATOR
 │
 ▼
WORKFLOW_WRITE_POST
 │
 ▼
Writer
 │
 ▼
Critic
 │
 ├── Status = REVISION_REQUIRED
 │        │
 │        ▼
 │   Writer (доработка по замечаниям Critic)
 │        │
 │        ▼
 │      Critic
 │
 ├── ... (цикл повторяется, максимум 5 итераций)
 │
 ├── Status = APPROVED
 │
 └── Достигнут MAX_ITERATIONS = 5
 │
 ▼
ORCHESTRATOR
 │
 ▼
OUTPUT_FORMAT
 │
 ▼
User
```

# Главные принципы архитектуры

- Один компонент — одна ответственность.
- ORCHESTRATOR управляет только запуском Workflow.
- WORKFLOW управляет процессом создания поста.
- Writer только создает и дорабатывает текст.
- Critic только проверяет текст.
- POST_STANDARD является единственным источником требований к качеству постов.
- Writer самостоятельно адаптирует текст под платформу, если она указана пользователем.
- Critic оценивает соответствие текста особенностям платформы, если она указана пользователем.
- Workflow завершается после получения статуса `APPROVED` либо достижения максимального количества итераций (`MAX_ITERATIONS = 5`). Основным критерием успешного завершения является статус `APPROVED`.
- OUTPUT_FORMAT только отображает готовый результат без каких-либо изменений.
