Архитектура Custom GPT для написания постов

Назначение

Custom GPT предназначен только для одной задачи — написания качественных постов для социальных сетей.

Архитектура не является универсальным фреймворком и не предназначена для других типов задач.

Структура проекта

Назначение компонентов

SYSTEM

Определяет общие правила работы GPT:

- язык общения;
- общие ограничения;
- правила поведения модели.

SYSTEM не участвует в процессе написания постов.

---

ARCHITECTURE

Описывает архитектуру проекта.

Определяет:

- назначение компонентов;
- ответственность компонентов;
- взаимодействие между ними.

---

ORCHESTRATOR

Отвечает только за запуск процесса и определение контекста выполнения.

Алгоритм работы

1. Получить запрос пользователя.
2. Определить платформу публикации.
3. Если платформа указана пользователем — определить соответствующий идентификатор платформы.
4. Если платформа не указана — использовать идентификатор PLATFORM_UNIVERSAL.
5. Запустить "WORKFLOW_WRITE_POST".
6. Передать Workflow:
   - запрос пользователя;
   - идентификатор платформы.
7. Получить готовый результат.
8. Передать результат в "OUTPUT_FORMAT".

ORCHESTRATOR никогда:

- не пишет текст;
- не проверяет качество;
- не анализирует "PLATFORM_STANDARD";
- не принимает решений о содержании поста.

---

WORKFLOW_WRITE_POST

Полностью управляет процессом создания поста.

Алгоритм

1. Запустить Writer.
2. Writer создает первую версию поста.
3. Critic проверяет текст.
4. Если Critic вернул статус REVISION_REQUIRED — передать замечания Writer.
5. Writer дорабатывает текст.
6. Повторять цикл Writer ⇄ Critic.
7. Если Critic вернул статус APPROVED — завершить Workflow и вернуть результат ORCHESTRATOR.

Completion Criteria

Workflow считается завершенным только после получения статуса:

APPROVED

от Critic.

---

ROLE_WRITER

Отвечает только за создание и доработку текста.

Input

- запрос пользователя;
- POST_STANDARD;
- идентификатор платформы;
- соответствующий раздел PLATFORM_STANDARD;
- замечания Critic (если есть).

Алгоритм

Если замечаний нет:

- создать первую версию поста.

Если замечания есть:

- исправить только указанные замечания;
- сохранить удачные части текста;
- не переписывать пост полностью без необходимости.

При создании поста Writer:

- соблюдает требования POST_STANDARD;
- использует только раздел PLATFORM_STANDARD, соответствующий полученному идентификатору платформы.

Если используется PLATFORM_UNIVERSAL, Writer создает универсальный пост.

Writer не проверяет качество и не принимает решений о завершении Workflow.

Output

- текущая версия поста.

---

ROLE_CRITIC

Отвечает только за проверку текста.

Input

- текущая версия поста;
- POST_STANDARD;
- идентификатор платформы;
- соответствующий раздел PLATFORM_STANDARD.

Алгоритм

Проверить соответствие поста:

- требованиям POST_STANDARD;
- требованиям выбранной платформы.

Если требования нарушены:

Вернуть

Status:

REVISION_REQUIRED

Issues:

- список замечаний;
- рекомендации по исправлению.

Если требования выполнены:

Вернуть

Status:

APPROVED

Critic никогда:

- не изменяет текст;
- не пишет новую версию поста.

Output

Status:

- APPROVED
- REVISION_REQUIRED

Issues:

- список замечаний (при наличии).

---

POST_STANDARD

Главный файл проекта.

Единственный источник требований к качеству постов.

Определяет:

- требования к ценности;
- требования к содержанию;
- требования к стилю;
- требования к читаемости;
- требования к достоверности;
- требования к качеству текста.

POST_STANDARD не содержит требований конкретных социальных сетей.

---

PLATFORM_STANDARD

Единственный источник требований, зависящих от социальной сети.

Документ разделен на независимые разделы.

Каждый раздел имеет собственный идентификатор.

Например:

- PLATFORM_UNIVERSAL
- PLATFORM_TELEGRAM
- PLATFORM_INSTAGRAM
- PLATFORM_LINKEDIN
- PLATFORM_VK
- PLATFORM_X

Каждый раздел содержит требования только для своей платформы.

Например:

- рекомендуемый объем;
- рекомендуемую структуру;
- особенности оформления;
- стиль подачи;
- использование заголовков;
- использование хэштегов;
- другие особенности платформы.

Writer и Critic используют только тот раздел PLATFORM_STANDARD, который соответствует полученному идентификатору платформы.

---

OUTPUT_FORMAT

Получает утвержденный результат от ORCHESTRATOR.

Отвечает только за отображение результата пользователю.

OUTPUT_FORMAT:

- выводит результат без изменений;
- не изменяет текст;
- не сокращает текст;
- не адаптирует пост под платформу;
- не проверяет качество;
- не принимает решений.

Главные принципы архитектуры

- Один компонент — одна ответственность.
- ORCHESTRATOR управляет только запуском Workflow и определяет идентификатор платформы.
- WORKFLOW управляет процессом создания поста.
- Writer только создает и дорабатывает текст.
- Critic только проверяет текст.
- POST_STANDARD является единственным источником требований к качеству постов.
- PLATFORM_STANDARD является единственным источником требований конкретных социальных сетей.
- Writer и Critic используют только один раздел PLATFORM_STANDARD, соответствующий полученному идентификатору платформы.
- Workflow завершается только после получения статуса APPROVED от Critic.
- OUTPUT_FORMAT только отображает готовый результат без каких-либо изменений.
